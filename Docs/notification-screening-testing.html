Notification Screening — setup & test guide

How to configure and manually verify the feature that screens third-party apps' notifications (SMS, voicemail, etc.) against SpamBlocker's own Number and Message rules, and dismisses the notification when the verdict is Block.

1What this feature does

SpamBlocker normally only screens calls and texts it receives directly. Many phones run a separate default Messages or Phone app that SpamBlocker never intercepts — this feature reads that other app's notifications instead, using the same Notification Access permission already granted for the older Push Alert feature. No new Android permission is required.

When an app in the App Notifications list posts a notification, its title and body are checked against every Number Rule and Message Rule that has Notification Title and/or Notification Body enabled under Apply to. If any rule's verdict is Block, the notification is dismissed immediately.

2One-time setup

Enable the feature

  1. Settings → Screening section → toggle Notifications on (sits directly below the SMS toggle — it works even if SMS screening is off).
  2. Grant Notification Access if prompted.
  3. Quick Settings → App Notifications → choose which apps to screen. The default Phone and SMS apps are pinned to the top of the picker.
Testing on an emulator: test notifications sent via adb shell cmd notification post are posted under the com.android.shell package identity, not a real app. Add com.android.shell to App Notifications so simulated test notifications actually get screened.

Create the five test rules

Two rules (Number 5551234567, Message .*test.*) cover the field-matching matrix in section 5. The remaining three, together with the first Number Rule, cover the priority/precedence matrix in section 6 — create all five up front.

  • Priority: 2
  • Apply to Notifications → Title, Body

  • Type: Block
  • Priority: 0
  • Apply to Notifications → Title, Body

.*blockfirst.*
  • Type: Block
  • Priority: 6
  • Apply to Notifications → Title, Body
Message Rule
.*allowme.*
  • Type: Allow
  • Priority: 3
  • Apply to Notifications → Title, Body
Message Rule
.*blocklast.*
  • Type: Block
  • Priority: 0
  • Apply to Notifications → Title, Body
Regex, not a literal word. Message Rules require the pattern to match the entire body/title. A bare test only matches a notification whose text is exactly "test" — write .*test.* to match anywhere in a sentence.
Priority. Higher priority number evaluates first; the first matching rule wins regardless of rule type or which field matched. Set priorities exactly as listed above — they're what the section 6 matrix exercises.

3Sending a test notification

Run from a PowerShell session with adb on PATH and the emulator connected:

$adb = "$env:LOCALAPPDATA\Android\Sdk\platform-tools\adb.exe"
& $adb shell "cmd notification post -t 'TITLE TEXT' 'unique_tag' 'BODY TEXT'"
Quote the whole command as one string. adb shell cmd notification post -t 'a' 'b' 'multi word body' passed as separate PowerShell arguments gets parsed twice — once by PowerShell, once by the remote shell — and silently truncates a multi-word body down to its first word. Always wrap the entire cmd notification post … invocation as a single quoted string handed to adb shell, exactly as shown above. Avoid parentheses in title/body text unless using this wrapped form.

4How to verify a test

Don't rely on shell tooling to check the result — inspect the emulator directly:

  • Pull down the notification shade — a Block verdict means the notification is gone or never appears.
  • Open SpamBlocker's SMS history tab — every screened notification appears as a row with its match reason (e.g. RegEx: 5551234567) or Passed by default if nothing matched.

6Priority / precedence test matrix

All 9 passed with no bugs found. Confirms priority ordering, cross-verdict tie-breaking (an Allow beating a lower-priority Block, and vice versa), and cross-rule-type precedence (Number vs. Message rule, title vs. body) all work correctly through the notification-screening path.
